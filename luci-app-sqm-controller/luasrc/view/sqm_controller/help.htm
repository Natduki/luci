系统概览

当前系统的工作链路可以理解为“接口整形 + 下载镜像 + 流量分类 + 策略决策 + 报告留痕”。上传方向直接在 WAN 接口上整形，下载方向先通过 ingress -> ifb0 重定向，再在 ifb0 上进行分类和统计。

01

基础整形

根据 basic_config 中的接口、上传带宽、下载带宽和队列算法建立 HTB + fq_codel/cake 基础树。

02

下载镜像

入口流量通过 ingress 规则重定向到 ifb0，下载方向的分类与统计依赖这条链路正常工作。

03

分类器

分类器读取 classification 和 class_rule，由防火墙打 mark，再由 tc 将 mark 映射到不同类。

04

策略与报告

策略引擎基于延迟、丢包和分类统计动态调整下载类；执行结果会写入状态文件与策略报告日志。

页面导览

基础设置

配置接口、上下行带宽、队列算法和日志级别，是所有运行链路的入口页面。

配置向导

用更短路径完成初始开局，适合快速设置带宽和启停状态。

状态监控

查看当前服务是否生效，确认 WAN 侧和 ifb0 侧是否已经建立 SQM 根 qdisc。

实时监控

观察时延、丢包和带宽变化，用于判断链路拥塞趋势和策略输入是否正常。

分类流量统计

查看 other / gaming / streaming / bulk 四类流量的实时速率与占比，验证分类是否命中。

场景模板

一键应用常见场景参数，适合演示和快速切换不同配置方案。

策略引擎

查看当前策略模式、判定原因、动作数量，并可手动执行一次策略判断。

策略报告

导出 JSON 或 CSV 报告，并预览最近的策略决策记录。

系统日志

查看后端日志、自检结果、轮转日志，定位服务未启动、分类失败和策略异常。

快速上手

  1. 选择接口并填写带宽 在“基础设置”中确认实际 WAN 接口，填写下载和上传带宽。数值是整形链路的基础，不建议留空或填明显超过实测值的数。
  2. 应用配置并启用服务 保存后启用服务，检查“状态监控”页面是否出现 1: 和 2: 根 qdisc。
  3. 验证下载链路进入 ifb0 打开“分类流量统计”，将设备切换到 ifb0，确认总速率在下载时能增长。
  4. 应用分类器 确保默认配置中的 classification 与 class_rule 存在，再执行分类器，使 mark 与 tc 类建立映射。
  5. 观察分类命中 触发网页视频、游戏或普通下载，检查 streaming、gaming、bulk 是否开始增长，而不是长期只有 other。
  6. 执行一次策略 在“策略引擎”中点击“执行一次策略”，确认策略状态、最近返回和报告导出功能均能正常工作。

建议

使用前先用“配置向导”完成一次基础初始化，再回到“基础设置”做细调，路径更短。

注意

下载方向分类依赖 ifb0。如果 ifb0 没有流量，再完善规则也不会命中。

排障

若页面一直显示无数据，先看服务状态、日志输出和基础 qdisc 是否已经建立，不要直接怀疑前端页面。

分类说明

当前项目默认使用四类语义：

other：其他流量 gaming：游戏流量 streaming：流媒体 bulk：批量传输
  • 规则来源：来自 /etc/config/sqm_controller 中的 classification 和多个 class_rule。
  • 优先级：优先级越高，规则越先参与归类；同一类中更具体的端口或 IP 条件优先级更高。
  • 端口匹配：当前规则支持 dport 和 sport。网页视频下载常见的是服务端源端口，适合用 sport。
  • 下载方向：下载流量只有在 ingress -> ifb0 重定向正常后，ifb0 上的类统计才会增长。
  • 统计验证：前端“分类流量统计”页面只是读当前类计数，命中与否应先从 tc 和日志判断，不要只盯页面刷新结果。

策略说明

  • 策略模式：支持自动模式和手动导向模式；自动模式会根据延迟、丢包和分类占比决定倾向。
  • 执行入口：页面的“执行一次策略”本质上调用后端一次性决策路径。
  • 状态快照：策略状态由 /tmp/sqm_policy_state.json 提供，页面会优先读取该快照。
  • 报告来源：策略报告来自 /var/log/sqm_policy.jsonl，导出页只是对该日志做展示和导出。
  • 动作数量：当页面显示无动作时，表示本次决策未触发类调整，不一定是系统故障。

配置与文件路径

/etc/config/sqm_controller 主配置文件，包含基础配置、分类配置、分类规则和策略参数。
/var/log/sqm_controller.log 系统运行日志，优先用于查看启动失败、分类失败、tc 命令失败等问题。
/var/log/sqm_policy.jsonl 策略报告日志，每次策略执行会追加一条记录，供报告页预览和导出。
/tmp/sqm_policy_state.json 当前策略状态快照，策略页用它展示最新模式和最近一次变更时间。
/usr/lib/sqm-controller/main.py 主要后端入口，负责状态、监控、自检、分类器、策略和报告等命令派发。
ifb0 下载镜像接口。下载方向的 tc 类和分类统计主要在该设备上完成。

常见排障

先检查“基础设置”中的启用状态和带宽是否有效，然后查看日志中是否存在 tc 命令失败。基础树未建立时，后续分类和策略都不会正常工作。

说明 ingress 重定向链路没有生效。优先检查 WAN 接口配置是否正确、ifb0 是否已创建并拉起，以及入口过滤器是否存在。

通常是规则条件不匹配。应结合抓包确认实际是 dport 还是 sport，并确认 mark 是否成功传递到下载侧 tc。

先确认 /var/log/sqm_policy.jsonl 是否存在且有内容。若策略从未成功执行，报告页自然无法展示记录。

前端页面只展示后端返回结果。应优先核对配置文件、后端日志、tc 输出和防火墙规则，不要把问题直接归因于页面渲染。

常用命令

python3 /usr/lib/sqm-controller/main.py --status-json
python3 /usr/lib/sqm-controller/main.py --monitor
python3 /usr/lib/sqm-controller/main.py --apply-classifier
python3 /usr/lib/sqm-controller/main.py --get-class-stats --dev ifb0
python3 /usr/lib/sqm-controller/main.py --policy-once
python3 /usr/lib/sqm-controller/main.py --export-report --format json

tc -s qdisc show dev eth0
tc -s class show dev ifb0
tc filter show dev ifb0 parent 2:
ip -s link show ifb0

cat /etc/config/sqm_controller
logread | grep sqm
tail -n 100 /var/log/sqm_controller.log

说明：命令中的接口名以你的实际 WAN 为准。若系统使用的是 eth1，请不要继续按 eth0 观察下载链路。
